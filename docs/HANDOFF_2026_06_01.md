## Key memory addresses (teleport-specific)

| Addr | What |
|------|------|
| FFBA | boss index 0-8 (Shalamar..Penta Dragon). Cycle counter when used by combo path. |
| FFBF | mini-boss flag. Always 0 before teleport (else 0x1A2B does RET NZ). |
| D880 | master scene state. 0x02=dungeon, 0x0C-0x14=boss arenas, 0x18=splash. **Recomputed every frame from FFBF/FFB7 by bank1:0x4F5D — do NOT write directly.** |
| FF93 | joypad state (active high, bit2=SELECT, bit3=START). Written by VBlank hook every frame. |
| FFFF | IE register. Critical for IRQ control during arena init. |
| DCBB | boss HP. We preset to 0x80 each teleport so boss doesn't instant-die → post-arena FFBA++ → cycle order garbage. |
| DCDC/DCDD | Sara HP sub/main. Preset to 0xFF each teleport. |

| WRAM | What we use it for |
|------|--------------------|
| DF0A | DX teleport request (Lua → ROM). 1-9 = boss + 1. Currently the v16 ROM ignores it. |
| DF0C | combo debounce (1 while combo held) |
| DF0E | landing-pad-copy sentinel (0x5A once copied) |
| DF1D | re-fire sit-out counter (30 frames) |
| DF1F | colorize-skip counter (60 frames after teleport so arena init can run uninterrupted) |
| DF20/21 | saved main-loop PC for the landing pad's `JP HL` back to game |
| **0xDB00** | **runtime landing pad (40 bytes copied from bank13:0x6F80). NOT formally verified game-safe — the leading hypothesis for the mgba v17 freeze.** |

## Suggested next steps (priority order)

### 1. Verify WRAM 0xDB00 is game-safe (the leading freeze hypothesis)

Run a long-form WRAM probe in mgba with a watchpoint on `0xDB00-0xDB27`
(the 40 bytes our landing pad occupies). Play 5-10 minutes of dungeon
+ arena gameplay. Any write means the game uses that range and our
landing pad gets corrupted → next teleport's `RETI` lands on garbage
→ freeze. The 24 extra bytes in v17 shift adjacent code by 24 bytes
which may coincidentally hit the corruption faster, explaining the
mgba-only-after-5-sec behavior.

If 0xDB00 IS used by the game: relocate the landing pad to a
formally-verified region. Candidates (need separate verification):
- The colorize handler's `DA00` block already has bg_table (256 bytes
  copied from ROM). The next 256 bytes (DB00-DBFF) is our current
  guess but unverified.
- Consider HRAM region 0xFFE0-0xFFFE if anywhere has 40 free bytes
  there. mgba's HRAM is fully addressable from the bank-13 hook.

### 2. Reproduce v17 freeze with logging

`/tmp/build_v17_dx_attempt.py` (if still on disk — moonshot only) has
the v17 builder. Save it to `scripts/build_v17_dx_attempt.py` first.
Then run with mgba's logging:
```
mgba-qt -l 65535 rom/working/penta_dragon_dx_teleport.gb 2> /tmp/mgba.log
```
The mode-3 write log is the likely smoking gun for hypothesis 2 (LCD
timing / OAM DMA).

### 3. Workaround: Lua-side combo simulation

If fixing the ROM is hard, the cleanest *useful* workaround is to
have the Lua bridge simulate the SELECT+START combo by:
- For "go to boss N": write `FFBA = N-2` (with the 0xFF wrap if N=0
  or 1), then `FF93 = 0x0C` for 1 frame, then 0 for ~30 frames.
- Our existing combo-path INC takes FFBA to N. Teleport.

This sidesteps the DF0A consumer entirely. It does require carefully
handling the 1-frame FF93 write so the colorize-handler's joypad read
sees it. Test in PyBoy first.

### 4. Re-verify boss-name garbling on splash

User noted some sprite glitching on the boss splash screen. We didn't
chase it. Likely related to forced `DCBB`/HP preset overriding game
state the splash expects to set up. Might just need to NOT preset HP
on splash entry (only on arena entry).

  not a bug.
